☁️ VPC 网络架构深度解析

核心要点：本文详细讲解云平台 VPC 网络架构的完整设计，包括虚拟交换机、路由器、NAT 网关、安全组等核心组件，以及虚机数据包的完整转发路径。

1. 架构概述

VPC (Virtual Private Cloud) 是云计算中最重要的网络基础设施。在我们的云平台中，VPC 采用三层分层架构：

  • 控制面：负责网络配置、策略下发、状态管理
  • 数据面：负责虚机流量转发、NAT 转换、安全过滤
  • 网关层：负责外网访问、负载均衡、流量汇聚
VPC 架构分层模型
┌──────────────────────────────���──────┐ │ 控制面 (Control Plane) │ │ - 配置管理 - 策略下发 - 监控 │ └─────────────────────────────────────┘ ↓ ┌─────────────────────────────────────┐ │ 数据面 (Data Plane) │ │ 虚拟交换机 → 路由器 → NAT → 安全组 │ └─────────────────────────────────────┘ ↓ ┌─────────────────────────────────────┐ │ 网关层 (Gateway Layer) │ │ - 外网网关 - 负载均衡 - EIP │ └─────────────────────────────────────┘

2. 三层分层设计

2.1 虚拟交换机（vSwitch）

虚拟交换机是虚机网络流量的第一跳，主要功能：

  • MAC 学习和转发
  • VLAN 隔离
  • 流量监控和限流
  • QoS 优先级处理
虚拟交换机实现方式
# 方案 1：基于 OVS (Open vSwitch) ovs-vsctl add-br br-vpc1 ovs-vsctl add-port br-vpc1 eth0 ovs-vsctl add-port br-vpc1 vnet12345 # 方案 2：基于 Linux Bridge brctl addbr br-vpc1 brctl addif br-vpc1 eth0 brctl addif br-vpc1 vnet12345 # 查看虚拟交换机信息 ovs-vsctl show ovs-ofctl dump-flows br-vpc1

2.2 虚拟路由器（vRouter）

虚拟路由器是网络流量的核心枢纽，实现：

  • 跨子网路由（三层转发）
  • NAT 转换（地址转换）
  • VIP 漂移（高可用）
  • QoS 和限流
虚拟路由器配置示例
# 创建虚拟路由器接口 ip link add vr-vpc1 type veth peer name vr-eth0 # 配置 IP 地址 ip addr add 10.0.1.1/24 dev vr-eth0 ip addr add 10.0.2.1/24 dev vr-eth1 # 启用 IP 转发 echo 1 > /proc/sys/net/ipv4/ip_forward # 添加静态路由 ip route add 10.0.0.0/16 via 192.168.100.1

2.3 NAT 网关

NAT 网关实现虚机与外网的通信，包括：

  • SNAT：虚机出站流量转换
  • DNAT：外网入站流量转换
  • EIP：弹性公网 IP 绑定
  • 连接跟踪：维护转换表
功能 源地址 目标地址 用途
SNAT 10.0.1.10 → 61.152.10.20 8.8.8.8:53 虚机访问外网
DNAT 8.8.8.8:53 61.152.10.20:80 → 10.0.1.100:80 外网访问虚机
EIP 61.152.10.20 10.0.1.100 绑定公网 IP

3. 虚机网络流程

虚机内部数据包的完整转发流程如下：

虚机发送数据包的完整流程
步骤 1：应用程序发送数据 (应用层) └─ 数据：TCP SYN 包 | 源IP: 10.0.1.10 | 目标IP: 10.0.2.20 步骤 2：虚机内核处理 (网络层) ├─ 查询路由表：目标 10.0.2.20 → 网关 10.0.1.1 ├─ ARP 解析：10.0.1.1 的 MAC 地址 └─ 添加 MAC 头：源 MAC | 目标 MAC 10.0.1.1 步骤 3：虚拟网卡输出 (vnet) └─ 数据帧：[MAC 头] [IP 头] [TCP 头] [数据] 步骤 4：虚拟交换机转发 (OVS/Bridge) ├─ MAC 学习：记录源 MAC 来自哪个端口 ├─ MAC 查表：查找���标 MAC 所在端口 └─ 转发到路由器接口 步骤 5：虚拟路由器处理 (L3) ├─ 检查目标 IP 10.0.2.20 ├─ 查询路由表：10.0.2.0/24 → 经由 vr-eth1 ├─ TTL 递减：TTL - 1 ├─ 重新封装： │ ├─ 新源 MAC：vr-eth1 MAC │ ├─ 新目标 MAC：虚机 10.0.2.20 MAC │ └─ 保持 IP 头和 TCP 头 └─ 转发到虚拟交换机 2 步骤 6：虚拟交换机 2 转发 (OVS/Bridge) └─ 转发数据帧到虚机 10.0.2.20 的虚网卡 步骤 7：虚机 2 接收数据 ├─ 虚网卡接收数据帧 ├─ 内核解析 MAC 头（匹配本机 MAC） ├─ 内核解析 IP 头（匹配本机 IP） ├─ 内核解析 TCP 头（匹配监听端口） └─ 将数据传给应用程序
⚠️ 重要：注意步骤 4 和 6 中，虚拟交换机在转发时需要学习 MAC 地址。如果长时间没有对向流量，MAC 表项会过期（通常 300 秒），导致转发失败。

4. 跨子网通信

跨子网通信必须经过虚拟路由器，流程如下：

  • 步骤 1-3：虚机 A 发送数据到虚拟交换机 1
  • 步骤 4：路由器接收，查询路由表，确定下一跳
  • 步骤 5-6：路由器转发到虚拟交换机 2
  • 步骤 7：虚机 B 接收数据
💡 优化建议：在路由器上启用 ECMP (等价多路径)，可以负载均衡多条链路，提高带宽利用率。

5. 外网访问流程

5.1 虚机出站（SNAT）

虚机出站流程（访问外网）
虚机 10.0.1.10 → DNS 查询 (8.8.8.8:53) 步骤 1：虚机发送数据 源 IP: 10.0.1.10 | 目标 IP: 8.8.8.8:53 步骤 2：虚拟交换机转发到路由器 步骤 3：路由器 SNAT 转换 ├─ 查询 NAT 规则：10.0.0.0/16 出站 → SNAT 到 61.152.10.20 ├─ 修改源 IP：10.0.1.10 → 61.152.10.20 ├─ 创建 conntrack 记录： │ ├─ 源：10.0.1.10:12345 │ └─ 转换后：61.152.10.20:12345 └─ 转发到外网网关 步骤 4：外网网关转发到 8.8.8.8 步骤 5：DNS 响应返回 源 IP: 8.8.8.8:53 | 目标 IP: 61.152.10.20:12345 步骤 6：路由器反向 SNAT（conntrack 查表） ├─ 查询 conntrack 记录 ├─ 目标 IP：61.152.10.20:12345 → 10.0.1.10:12345 └─ 转发回虚机 10.0.1.10

5.2 外网入站（DNAT）

外网用户访问虚机的 Web 服务：

外网入站流程（DNAT）
外网用户 → 访问 http://61.152.10.20:80 步骤 1：外网用户发送请求 源 IP: 1.1.1.1 | 目标 IP: 61.152.10.20:80 步骤 2：路由器接收 DNAT 规则 ├─ 规则：61.152.10.20:80 → 10.0.1.100:80 ├─ 修改目标 IP：61.152.10.20 → 10.0.1.100 ├─ 创建 conntrack 记录 └─ 转发到虚机 10.0.1.100 步骤 3：虚机 Web 服务处理请求并响应 步骤 4：虚机响应数据返回 源 IP: 10.0.1.100:80 | 目标 IP: 1.1.1.1 步骤 5：路由器反向 DNAT（conntrack 查表） ├─ 修改源 IP：10.0.1.100 → 61.152.10.20 └─ 转发给外网用户 1.1.1.1 步骤 6：外网用户接收响应 源 IP: 61.152.10.20:80 | 目标 IP: 1.1.1.1

6. 故障排查

6.1 虚机无法访问外网

排查步骤（5 层诊断）
【第 1 层】物理网络 $ ethtool eth0 | grep "Link detected" Link detected: yes ✓ 网卡正常 【第 2 层】虚拟网卡 $ ip link show vnet12345 ... LOWER_UP ... ✓ 虚拟网卡正常 【第 3 层】虚拟交换机 $ ovs-ofctl dump-flows br-vpc1 | grep vnet12345 ... ✓ OVS 规则正常 【第 4 层】路由表和 ARP $ ip route show default via 10.0.1.1 dev eth0 ✓ 路由正确 $ arp -n | grep 10.0.1.1 ... 00:11:22:33:44:55 ✓ ARP 解析正确 【第 5 层】NAT 和防火墙 $ iptables -L -n -t nat | grep 10.0.1.10 ... SNAT all -- 10.0.0.0/16 0.0.0.0/0 to:61.152.10.20 ✓ NAT 规则正确 $ cat /proc/net/nf_conntrack | grep 10.0.1.10 ... (查看连接跟踪表) ✓ conntrack 正常

6.2 常见问题

问题 症状 原因 解决方案
无法 ping 网关 网络不可达 虚拟交换机故障 检查 OVS/Bridge 配置
能 ping 网关但上不了外网 DNS 超时 NAT 规则缺失 添加 SNAT 规则
DNS 偶发超时 间歇性无法解析 conntrack 表满 增加 nf_conntrack_max
返程流量丢失 半连接状态 反向 NAT 失败 检查 conntrack 表项

7. 性能优化

7.1 OVS 流表优化

OVS 流表优化配置
# 增加流表大小 ovs-vsctl set Bridge br-vpc1 flow_tables:0=@ft ovs-vsctl --no-headings --columns _uuid find Flow_Table # 启用 DPDK（性能提升 10 倍） ovs-vsctl set Open_vSwitch . other_config:dpdk-init=true # 调整缓存参数 ovs-vsctl set Open_vSwitch . other_config:n-dpdk-rxqs=4 # 查看流表统计 ovs-ofctl dump-stats br-vpc1

7.2 conntrack 优化

conntrack 参数调优
# 增加 conntrack 表大小（默认 65536） echo 1000000 > /proc/sys/net/nf_conntrack_max # 启用 conntrack 复用 echo 1 > /proc/sys/net/ipv4/tcp_tw_reuse # 减少超时时间（加快连接释放） echo 300 > /proc/sys/net/ipv4/netfilter/ip_conntrack_tcp_timeout_established # 监控 conntrack 使用情况 wc -l /proc/net/nf_conntrack

8. 最佳实践

✓ 最佳实践清单：
  • 使用虚拟路由器的 ECMP 实现负载均衡
  • 定期监控 conntrack 表占用率
  • 启用 OVS DPDK 提高性能
  • 配置完整的监控和告警体系
  • 定期清理过期的 MAC 表项
  • 使用安全组精细化控制流量
