🏗️ 高可用架构设计

文章摘要：本文从高可用等级定义出发，详细讲解应用层、数据层、网络层的多层冗余设计，重点分析故障检测与转移的时间指标（RTO/RPO），以及如何通过仲裁机制防止脑裂。包含完整的无感知升级方案和真实的级联故障案例分析。该方案已在云平台支撑 50+ 租户、10000+ 虚机的高可用保障。

1. 高可用等级划分

1.1 SLA 等级与可用性指标

高可用等级由 SLA（Service Level Agreement）定义，用九的个数表示可用性：

等级 可用性 每年停机时间 每月停机时间 典型应用
2 个 9 99.00% 87.6 小时 7.2 小时 内部系统、开发环境
3 个 9 99.90% 8.76 小时 43.2 分钟 普通互联网服务
4 个 9 99.99% 52.6 分钟 4.3 分钟 金融、电商、云计算
5 个 9 99.999% 5.26 分钟 26 秒 超大规模互联网、运营商

1.2 RTO 和 RPO 指标

两个最重要的可用性指标：

RTO vs RPO 对比
RTO（Recovery Time Objective - 恢复时间目标） ┌─────────────────────────────────────────────┐ │ 定义：系统发生故障到恢复的时间 │ │ 单位：秒、分钟、小时 │ │ 例子：主数据库宕机到备库接管 = 3 秒 │ │ 影响：业务中断时间长度 │ └─────────────────────────────────────────────┘ 故障发生 ────────────────────────── 系统恢复 ╰──── RTO（< 3s 目标） ──╯ RTO = 3 秒意味着： - 故障检测：1 秒 - VIP 漂移：1 秒 - 应用重连：1 秒 总计：3 秒内客户端能连接到新 Master RPO（Recovery Point Objective - 恢复点目标） ┌─────────────────────────────────────────────┐ │ 定义：故障时可接受的最大数据丢失量 │ │ 单位：秒、分钟、小时 │ │ 例子：主库到备库的复制延迟 = 500ms │ │ 影响：可能丢失的数据量 │ └─────────────────────────────────────────────┘ 最后一次数据同步 ────────────────── 故障发生 ╰── RPO(< 500ms) ──╯ RPO = 500ms 意味着： - 最多丢失 500ms 的数据 - 需要同步复制或接近同步复制 【实际案例】支付系统需求： - RTO < 3 秒（用户感知：连接超时重试） - RPO = 0（绝对不能丢单笔交易） - 解决方案：同步复制（牺牲性能换数据安全） 【实际案例】日志系统需求： - RTO < 30 秒（可接受短暂不可用） - RPO < 1 分钟（允许丢失少量日志） - 解决方案：异步复制（保证性能）

【主库配置】Master-DB (192.168.1.10) # my.cnf [mysqld] # 复制配置 server-id = 1 log_bin = mysql-bin binlog_format = ROW # 行级日志，更安全 binlog_row_image = FULL # 同步复制配置（关键） rpl_semi_sync_master_enabled = ON # 启用半同步复制 rpl_semi_sync_master_wait_for_slave_count = 1 # 等待至少 1 个从库确认 rpl_semi_sync_master_timeout = 10000 # 超时 10s 后退化为异步 # 写优化 sync_binlog = 1 # 每个事务都同步到磁盘（安全但慢） innodb_flush_log_at_trx_commit = 1 # 每个事务都 fsync # 性能监控 slow_query_log = ON slow_query_log_file = /var/log/mysql/slow.log long_query_time = 0.1 # 100ms 以上的查询记录 【从库配置】Slave-DB (192.168.1.11) [mysqld] server-id = 2 relay-log = mysql-relay-bin read_only = ON # 从库只读 # 同步复制配置 rpl_semi_sync_slave_enabled = ON # 复制线程优化 slave_parallel_workers = 4 # 4 个并行复制线程 slave_parallel_type = LOGICAL_CLOCK # 基于逻辑时钟并行 # 性能优化 skip-slave-start # 防止自动启动复制（管理员控制） 【监控脚本】检查复制延迟 #!/bin/bash SLAVE_HOST="192.168.1.11" SLAVE_USER="root" SLAVE_PASS="password" SECONDS_BEHIND=$(mysql -h $SLAVE_HOST -u $SLAVE_USER -p$SLAVE_PASS \ -e "SHOW SLAVE STATUS\G" | grep "Seconds_Behind_Master" | awk '{print $2}') ALERT_THRESHOLD=1 # 延迟超过 1 秒告警 if [ "$SECONDS_BEHIND" = "NULL" ]; then echo "CRITICAL: Slave is not running" exit 2 elif [ "$SECONDS_BEHIND" -gt "$ALERT_THRESHOLD" ]; then echo "WARNING: Replication lag ${SECONDS_BEHIND}s (threshold: ${ALERT_THRESHOLD}s)" exit 1 else echo "OK: Replication lag ${SECONDS_BEHIND}s" exit 0 fi 【故障转移脚本】从库升级为主库 #!/bin/bash SLAVE_HOST="192.168.1.11" SLAVE_USER="root" SLAVE_PASS="password" # 1. 停止从库复制 mysql -h $SLAVE_HOST -u $SLAVE_USER -p$SLAVE_PASS \ -e "STOP SLAVE;" # 2. 等待日志应用完毕 mysql -h $SLAVE_HOST -u $SLAVE_USER -p$SLAVE_PASS \ -e "SELECT MASTER_POS_WAIT(@@GLOBAL.binlog_file, @@GLOBAL.binlog_position, 10);" # 3. 提升为主库（最关键） mysql -h $SLAVE_HOST -u $SLAVE_USER -p$SLAVE_PASS \ -e "RESET SLAVE ALL;" # 4. 验证 mysql -h $SLAVE_HOST -u $SLAVE_USER -p$SLAVE_PASS \ -e "SHOW MASTER STATUS\G" echo "Slave promoted to Master" 【应用层连接切换】 原配置： db_host = 192.168.1.10 # 固定主库地址 db_port = 3306 db_user = root db_pass = password 改进配置（推荐）： # 使用 VIP，由 keepalived 管理 db_host = 192.168.1.100 # 虚拟 IP（主库地址） db_port = 3306 db_user = root db_pass = password # keepalived 在主库故障时自动切换该 VIP 到从库 好处： - 应用无需修改 - 故障转移透明 - RTO < 5 秒

3. 故障检测与转移

3.1 故障检测时间分析

完整的故障检测时间链
【场景】主应用服务器网卡故障 T+0s 应用正常运行 ├─ Keepalived 运行 ├─ VRRP 心跳发送：Master → Backup └─ VIP: 10.0.1.1 在 Master 上 T+0.5s ⚠️ 网卡故障（例如光纤断裂） ├─ 物理网卡 DOWN ├─ Keepalived 进程仍运行（但网络已断） └─ Master 认为自己还活着 T+1s 第 1 个 VRRP 报文应该发送，但网卡故障导致失败 ├─ Backup 未收到报文 └─ Backup 启动计时器 T+2s 第 2 个 VRRP 报文（应该在 T+1s）未到达 ├─ Backup 记录：已丢失 1 个报文 └─ 计时器继续 T+3s 第 3 个 VRRP 报文（应该在 T+2s）未到达 ├─ Backup 记录：已丢失 2 个报文 └─ 触发多路径检测脚本 T+4s 第 3 次 VRRP 报文未到达 ├─ Backup 记录：已丢失 3 个报文（触发故障判定） ├─ 多路径检测脚本确认： │ ├─ Gateway ping: FAIL │ ├─ Master ICMP: FAIL │ ├─ Master TCP: FAIL │ └─ 决策：Master 真的故障，安全转移 │ ├─ 🔄 Backup 状态转移：BACKUP → MASTER ├─ 🔗 配置 VIP：ip addr add 10.0.1.1 ├─ 📡 发送无偿 ARP（3 次） ├─ ✓ Keepalived 执行 notify_master 脚本 └─ 🎯 VIP 激活完成 T+4.5s VIP 激活，但客户端 ARP 缓存还未更新 T+5s 客户端 ARP 缓存更新 ├─ 新的 VRRP 报文来自新 Master ├─ 所有客户端更新 ARP 缓存：10.0.1.1 → 新 MAC └─ 流量开始流向新 Master ┌────────────────────────────────────────┐ │ 完整故障转移时间统计 │ ├────────────────────────────────────────┤ │ 故障检测：0.5s（网卡故障发生） │ │ VRRP 超时：3s（3 × 1s 报文间隔） │ │ VIP 配置：0.1s │ │ ARP 更新：0.5s │ │ 应用重连：0.5s（取决于应用超时） │ ├────────────────────────────────────────┤ │ 总 RTO：~4-5 秒 │ └────────────────────────────────────────┘ 【优化方案】降低 RTO 1. 降低 advert_int（VRRP 心跳间隔） 原配置：advert_int = 1s → 故障检测 = 3s 优化配置：advert_int = 200ms → 故障检测 = 600ms 权衡：心跳网络开销 = 原来的 5 倍 2. 增加多路径检测频率 原配置：检测间隔 = 2s 优化配置：检测间隔 = 500ms 权衡：CPU 开销增加 3. 提前配置冗余路径 使用主机间直连的网卡（不走交换机） 确保至少有 1 条网络路径总是通的 【实际测试数据】 配置组合 A（保守）： - advert_int = 1s - multi-path interval = 2s - 结果：RTO = 4-5s，CPU < 1% 配置组合 B（均衡）： - advert_int = 500ms - multi-path interval = 1s - 结果：RTO = 2-3s，CPU ≈ 2% 配置组合 C（激进）： - advert_int = 100ms - multi-path interval = 200ms - 结果：RTO = 500ms-1s，CPU ≈ 5% 推荐：配置组合 B（均衡，适合大多数业务）

4. 脑裂与仲裁机制

4.1 脑裂原理回顾

当网络分割导致 Master 和 Backup 无法通信时，两边都会变成 MASTER，拥有 VIP，造成数据混乱。

脑裂发生的典型场景
【场景 1】交换机故障导致完全分割 正常状态： Master (10.0.1.10) ────────────── Backup (10.0.1.11) │ │ └──────────────┬───────────────┘ 交换机 (正常) 网络分割： Master (10.0.1.10) 交换机故障 Backup (10.0.1.11) │ (中断) │ └─── 隔离 ───────────────────────── 隔离 ───┘ 后续行为： Master 看不到 Backup: • 3 次心跳超时后判定 Backup 故障 • Master 保持或确认自己是 Master • 继续拥有 VIP，转发流量 Backup 看不到 Master: • 3 次心跳超时后判定 Master 故障 • Backup 认为自己应该变成 Master • 配置 VIP，开始转发流量 ❌ 结果：双 Master 脑裂！ 【场景 2】级联故障（最隐蔽） 时间线： T+0s: 主数据库故障 ├─ Master App 无法连接数据库 ├─ 健康检查失败 └─ Priority 降低：100 → 50 T+0.5s: 应用虽然故障但 Keepalived 进程仍活着 ├─ VRRP 心跳继续发送 ├─ Backup 收到心跳，知道 Master 还活着 └─ Backup 保持 Backup 状态 T+1s: 此时 Backup 的应用也故障了（级联） ├─ Backup 优先级：90 > Master 优先级 50 ├─ Backup 优先级更高！ └─ Backup 应该变成 Master T+1.5s: Backup 变成 Master（现在有 VIP） ├─ 原 Master Priority = 50（应用故障状态） ├─ 新 Master Priority = 90（应用仍故障！） └─ ⚠️ 应用故障但有 VIP，会导致业务瘫痪 ❌ 结果：接管故障应用，没有改善业务

4.2 仲裁机制：防脑裂的关键

三节点仲裁架构
【问题】两节点 Keepalived 无法判断谁真的故障 【解决】引入第三方仲裁节点 架构： ┌─────────────────────────────────────┐ │ 仲裁节点 (Arbiter) │ │ 轻量级，可以是一台 1GB 内存的主机 │ │ 或容器化部署 │ │ 职责：投票决定 VIP 归属 │ └─────────────────────────────────────┘ ↑ ← 心跳 ↑ ← 心跳 │ │ ┌────┴────┐ ┌────┴────┐ │ Master │ │ Backup │ │ │◄────►│ │ │ Vote: ?│ VIP?│Vote: ? │ └─────────┘ └─────────┘ 【决策逻辑】仲裁规则 if Master 和 Backup 都能连接仲裁节点： 仲裁节点投票给 Priority 更高的节点 结果：Master 或 Backup，唯一 if Master 能连接仲裁节点，Backup 不能： 仲裁节点投票给 Master 结果：Master 获胜，继续拥有 VIP if Backup 能连接仲裁节点，Master 不能： 仲裁节点投票给 Backup 结果：Backup 接管，变成 Master ✓ if 都不能连接仲裁节点： Master 继续持有 VIP（保守选择） 或两个都释放 VIP（激进选择） 【实现方式】基于 TCP 连接 #!/bin/bash # 仲裁节点（运行在第三台机器） # nc -l 0.0.0.0 9999 << EOF # VOTE_FOR_MASTER # EOF # Master 和 Backup 都尝试连接仲裁节点 ARBITER_HOST="192.168.1.100" ARBITER_PORT="9999" # 连接成功 = 仲裁节点确认该节点可见 if nc -zv $ARBITER_HOST $ARBITER_PORT 2>&1 | grep -q "succeeded"; then # 该节点能被仲裁节点看见 # 即使看不见另一端，仲裁节点也投票给该节点 echo "ARBITRATION_WON" else # 该节点看不见仲裁节点 # 网络可能分割，保守降低优先级 echo "ARBITRATION_LOST" fi 【集成到 Keepalived】 vrrp_script check_arbitration { script "/usr/local/bin/check_arbiter.sh 192.168.1.100 9999" interval 1 weight -100 # 仲裁失败时大幅降低优先级 fall 2 # 连续失败 2 次 } vrrp_instance VPC_VIP_1 { priority 100 track_script { check_split_brain # 多路径检测 check_arbitration # 仲裁检测 } } 【性能对比】 无仲裁： - 脑裂风险：高（10% 脑裂率） - 故障检测：快（3 秒） - 监控告警：困难 有仲裁： - 脑裂风险：极低（< 0.1%） - 故障检测：稍慢（4 秒，含仲裁验证） - 监控告警：简单（仲裁节点状态即可判断） 推荐：关键业务必须部署仲裁

5. 无感知升级方案

5.1 灰度升级流程

应用版本升级的无停机方案
【目标】升级应用版本，不中断客户端连接 当前状态： Master-1 (v1.0.0) Master-2 (v1.0.0) Master-3 (v1.0.0) 处理 30%流量 处理 35%流量 处理 35%流量 VIP: 10.0.1.1 VIP: 10.0.2.1 VIP: 10.0.3.1 【步骤 1】准备新版本 (v1.1.0) 在新节点上构建和测试新版本： $ cd /opt/app $ git clone ... v1.1.0 $ make build $ make test $ docker build -t app:1.1.0 . 新版本必须通过： - 单元测试 - 集成测试 - 灰度环境完整测试 【步骤 2】灰度发布到 Master-1 流程： 1. 选择 Master-1 作为灰度版本的首批承载 2. 健康检查器：暂停对 Master-1 的流量 在 LB 上执行： $ ipvsadm -e -t 10.0.1.1:80 -r 10.0.1.10:80 -w 0 这样做： - 现有连接继续处理（不中断） - 新连接不会路由到 Master-1 - 等待现有连接关闭（通常 < 1 分钟） 3. 停止旧版本应用 $ systemctl stop app-v1.0.0 确保： - 等待长连接断开 - 清理会话（存储到 Redis） - 关闭数据库连接 4. 启动新版本应用 $ systemctl start app-v1.1.0 验证： - 应用进程正常运行 - 连接到数据库成功 - 健康检查通过 5. 恢复流量权重 $ ipvsadm -e -t 10.0.1.1:80 -r 10.0.1.10:80 -w 100 现在 Master-1 开始处理新流量（v1.1.0） 6. 监控和验证 - 监控错误率：应该保持不变或下降 - 监控响应时间：应该接近或更快 - 检查日志：没有异常 ┌─ 灰度阶段 ──────────────────────────────┐ │ │ │ Master-1 (v1.1.0) - 10% 流量 [GRAY] │ │ Master-2 (v1.0.0) - 45% 流量 │ │ Master-3 (v1.0.0) - 45% 流量 │ │ │ │ 监控时间：15-30 分钟 │ │ 关键指标： │ │ - 错误率 < 0.1% │ │ - P95 延迟 < 预期 │ │ - CPU/内存 < 阈值 │ │ │ └──────────────────────────────────────────┘ 【步骤 3】扩大灰度范围到 Master-2 重复步骤 2 的过程： 1. 暂停 Master-2 流量 2. 等待连接关闭 3. 停止 v1.0.0，启动 v1.1.0 4. 恢复流量 现在状态： Master-1 (v1.1.0) - 50% 流量 Master-2 (v1.1.0) - 50% 流量 [GRAY] Master-3 (v1.0.0) - 0% 流量 再次监控 15 分钟 【步骤 4】升级最后一个节点 Master-3 重复步骤 2 的过程。 最终状态： Master-1 (v1.1.0) ✓ Master-2 (v1.1.0) ✓ Master-3 (v1.1.0) ✓ 升级完成！ 【完整脚本】自动化升级流程 #!/bin/bash TARGET_NODE="10.0.1.10" NEW_VERSION="1.1.0" VIP="10.0.1.1" SERVICE="app" # 步骤 1：暂停流量 echo "Pausing traffic to $TARGET_NODE..." ipvsadm -e -t ${VIP}:80 -r ${TARGET_NODE}:80 -w 0 # 步骤 2：等待连接清空 echo "Waiting for active connections to close..." sleep 60 # 验证连接已关闭 ACTIVE_CONN=$(ss -ant | grep :80 | grep $TARGET_NODE | wc -l) if [ $ACTIVE_CONN -gt 0 ]; 6. 级联故障防护

6.1 级联故障的定义和风险

级联故障（Cascading Failure）是指某个组件故障导致其他组件也故障，进而整个系统崩溃的现象。

级联故障的实际案例分析
【案例】2024 年 3 月，某云平台故障 初始故障：主数据库磁盘满 T+0s: 主 DB 磁盘 95% 满 ├─ 无法写入新日志 └─ 但读操作仍可 T+30s: 应用集群检测到数据库写入超时 ├─ 超时时间：30 秒 ├─ 应用线程卡住（等待数据库响应） └─ 应用仍能处理读请求 T+1m: 应用线程池耗尽 ├─ 所有线程都在等待数据库 ├─ 新请求无线程处理 ├─ 应用开始拒绝新连接 └─ HTTP 503 Service Unavailable T+1.5m: 客户端立即转向备节点 ├─ 备节点收到所有流量 ├─ 备节点也无法写数据库（同一个主 DB） └─ 备节点线程池也耗尽 T+2m: 所有节点都故障 ├─ 级联失败：1 个组件 → 全部故障 ├─ 用户无法访问 └─ ❌ 高可用完全失效 故障链： 磁盘满 → DB 写入超时 → 应用超时 → 线程池耗尽 → 全部故障 ❌ 高可用架构无能为力！ 【问题分析】为什么高可用失效 原因 1：单点故障（所有节点连同一个 DB） 原因 2：应用没有超时断路（等待无限期） 原因 3：没有熔断器（故障不隔离） 原因 4：没有流量控制（故障节点继续接收新请求） 【解决方案】断路器 + 熔断 + 降级 1. 数据库分片 ├─ 不是所有应用连同一个 DB ├─ 按租户或业务分片 └─ 一个 DB 故障只影响部分用户 2. 快速超时（关键！） 原配置：db_connect_timeout = 30s 优化配置：db_connect_timeout = 2s 好处： - 应用快速失败，不卡住 - 线程可以释放处理其他请求 - 故障隔离在数据库层 3. 断路器（Circuit Breaker） 状态机： CLOSED (正常) → 连续错误 > 5 个 → OPEN (断开) ↓ 30s 后 → HALF_OPEN ↓ 尝试 1 个请求 ↓ 成功 → CLOSED 失败 → OPEN 实现： class DBCircuitBreaker: def __init__(self): self.state = "CLOSED" self.error_count = 0 self.last_open_time = None def call(self, db_operation): if self.state == "OPEN": if time.time() - self.last_open_time > 30: self.state = "HALF_OPEN" else: # 断路器打开，直接返回错误，不调用 DB raise CircuitBreakerOpenException() try: result = db_operation() self.error_count = 0 self.state = "CLOSED" return result except Exception as e: self.error_count += 1 if self.error_count > 5: self.state = "OPEN" self.last_open_time = time.time() raise 4. 服务降级 如果 DB 故障，应用可以： ├─ 返回缓存数据（过期但可用） ├─ 返回默认值（部分功能） └─ 返回 "服务维护中" 提示 示例： try: user = db.query_user(user_id) except CircuitBreakerOpenException: # DB 故障，从缓存读 user = cache.get(user_id, default={ 'id': user_id, 'name': 'Unknown', 'cached': True }) log.warning(f"Using cached user data for {user_id}") 5. 流量控制（Rate Limiting） 即使故障，也不应该让流量打垮系统 实现： - 漏桶算法：固定流量 - 令牌桶算法：可突发 - 滑动窗口：精确控制 示例配置： max_connections = 100 # 最大并发连接 max_requests_per_second = 1000 # 最大请求速率 max_queue_size = 500 # 等待队列大小 超过限制时： ├─ 拒绝新请求（返回 429） ├─ 不让故障扩散 └─ 保护系统自身 【完整的级联故障防护示例】 #!/usr/bin/env python3 from enum import Enum from datetime import datetime, timedelta import time import random class CircuitBreakerState(Enum): CLOSED = "CLOSED" OPEN = "OPEN" HALF_OPEN = "HALF_OPEN" class ResilientDatabase: def __init__(self, db_config): self.db_config = db_config self.circuit_breaker_state = CircuitBreakerState.CLOSED self.error_count = 0 self.last_open_time = None self.error_threshold = 5 self.recovery_timeout = 30 self.db_timeout = 2 # 快速超时 self.cache = {} def query(self, query_str): """执行数据库查询，带级联故障防护""" # 步骤 1：检查断路器状态 if self.circuit_breaker_state == CircuitBreakerState.OPEN: if time.time() - self.last_open_time > self.recovery_timeout: # 尝试恢复 self.circuit_breaker_state = CircuitBreakerState.HALF_OPEN print(f"[CB] Transitioned to HALF_OPEN after {self.recovery_timeout}s") else: # 断路器仍打开 print(f"[CB] Circuit breaker is OPEN, using fallback") return self._fallback(query_str) # 步骤 2：执行查询（超时保护） try: result = self._execute_query(query_str) # 步骤 3：查询成功，重置错误计数 self.error_count = 0 self.circuit_breaker_state = CircuitBreakerState.CLOSED print(f"[OK] Query succeeded, CB state = CLOSED") return result except TimeoutError: # 步骤 4：查询失败（超时） self.error_count += 1 print(f"[ERR] DB timeout, error_count = {self.error_count}") if self.error_count >= self.error_threshold: # 触发断路器 self.circuit_breaker_state = CircuitBreakerState.OPEN self.last_open_time = time.time() print(f"[CB] Circuit breaker OPENED after {self.error_count} errors") # 使用降级方案 return self._fallback(query_str) def _execute_query(self, query_str): """实际执行查询（模拟）""" # 模拟随机超时 if random.random() < 0.7: # 70% 概率超时 raise TimeoutError("DB connection timeout") # 正常情况下返回结果 return {"result": "success"} def _fallback(self, query_str): """降级方案：返回缓存或默认值""" print(f"[FALLBACK] Using cached or default data") # 尝试从缓存读 if query_str in self.cache: print(f"[CACHE HIT] Returning cached data") return self.cache[query_str] # 返回默认值 print(f"[DEFAULT] Returning default value") return {"result": "default", "cached": True} # 测试 print("=== Cascading Failure Protection Demo ===\n") db = ResilientDatabase({}) for i in range(15): print(f"\n[Request {i+1}]") result = db.query("SELECT * FROM users") print(f"Result: {result}") time.sleep(0.5) 【监控和告警】 关键指标： 1. 断路器状态（OPEN/CLOSED/HALF_OPEN） - 告警：频繁转换（> 3 次/分钟） 2. 错误率（Error Rate） - 告警：> 1%（立即告警） - 严重：> 10%（页面告警） 3. 响应时间（Latency） - 告警：P95 > 5s（可能有故障） - 严重：P95 > 30s（级联故障迹象） 4. 活跃连接数 - 告警：> 80% 最大连接数 - 严重：连接数持续增长（连接泄漏）

案例：级联故障的完整复盘
【事件】2024 年 4 月 15 日，10:23 AM，某云平台故障 == 故障时间线 == 10:23:00 主数据库磁盘告急（85%） 日志系统告警（未及时处理） 10:23:30 慢查询增加，DB 响应变慢 应用开始出现超时错误（< 0.1%） 告警：Database latency > 1s 10:24:00 DB 磁盘已 92%，写入性能急剧下降 应用超时错误增加到 5% 开发: "是不是网络问题？" 10:24:30 DB 磁盘已 98%，几乎无法写入 应用线程开始卡住（等待 DB 响应） 用户开始反馈："页面加载很慢" 10:25:00 应用线程池耗尽（100 个线程全卡住） 所有请求返回 503 用户："网站完全访问不了" 10:25:15 LB 检测到 App 节点 1 无响应 自动从 LB 中移除 所有流量转向 App 节点 2 和 3 10:25:30 App 节点 2 和 3 也开始返回 503 因为它们也连同一个数据库（单点） 级联故障：所有应用节点故障 10:25:45 SLA 告警触发 Pagerduty 页面告警（收到） 管理员开始紧急响应 10:26:30 管理员发现：DB 磁盘满 操作： 1. 清理 binlog（释放 50GB） 2. 重启 MySQL 10:27:00 MySQL 重启成功 应用恢复，线程池释放 所有请求逐渐恢复正常 10:28:00 系统完全恢复 == 故障统计 == 总故障时间：5 分钟 影响用户：100% 数据丢失：0（DB 没崩） SLA 影响：4 个 9 的 SLA 违约（> 4 分钟） == 根本原因分析（RCA） == 1. 直接原因：磁盘满 ├─ 日志没有定期清理 ├─ binlog 没有自动清理 └─ 监控告警没有及时处理 2. 传播原因：应用连接池超时配置不当 ├─ 超时时间：默认 30 秒（太长） ├─ 应该：2-5 秒 └─ 后果：线程卡住，无法快速失败 3. 架构原因：单点数据库 ├─ 所有应用连同一个 DB ├─ 应该：多个 DB 实例（读写分离） └─ 后果：一个 DB 故障影响全部应用 4. 监控原因：告警阈值设置过高 ├─ 磁盘告警：85%（太晚了） ├─ 应该：70-75% └─ 后果：没有充足时间处理 5. 级联防护缺失：无断路器和熔断 ├─ 应该：在应用层实现 Circuit Breaker ├─ 效果：快速失败，降级处理 └─ 后果：级联到所有应用节点 == 改进方案 == 1. 磁盘管理（立即实施） #!/bin/bash # 自动清理日志和 binlog ALERT_THRESHOLD=75 # 磁盘使用 75% 时告警 ACTION_THRESHOLD=80 # 80% 时自动清理 USAGE=$(df /var/lib/mysql | awk 'NR==2 {print int($5)}') if [ $USAGE -ge $ACTION_THRESHOLD ]; then # 自动清理 binlog mysql -e "PURGE BINARY LOGS BEFORE DATE_SUB(NOW(), INTERVAL 7 DAY);" # 清理应用日志 find /var/log -name "*.log" -mtime +30 -delete logger "Auto cleanup: freed $((USAGE - $(df /var/lib/mysql | awk 'NR==2 {print int($5)}'))% disk" fi 2. 数据库连接超时优化 原配置： connection_timeout = 30s # 太长！ 优化配置： connection_timeout = 2s # 快速超时 query_timeout = 3s # 查询超时 max_retries = 3 # 最多重试 3 次 效果：DB 故障时，应用在 2 秒内快速失败，释放线程 3. 应用层断路器（Circuit Breaker） class DatabaseConnection: def __init__(self): self.circuit_breaker = CircuitBreaker( failure_threshold=5, recovery_timeout=30, timeout=2.0 ) def query(self, sql): @self.circuit_breaker.call def _query(): return self.db.execute(sql) return _query() 4. 数据库分片（架构改进） 原架构： App 1, 2, 3 -> 单一主 DB 改进架构： App 1 -> DB Shard 1 (读写分离) App 2 -> DB Shard 2 (读写分离) App 3 -> DB Shard 3 (读写分离) 好处： - 一个 Shard 故障只影响 1/3 用户 - 性能提升：QPS 提升 3 倍 - 扩展性：可以继续增加 Shard 5. 监控告警优化 磁盘监控： ├─ 黄色告警：70%（提前预警） ├─ 橙色告警：80%（紧急处理） └─ 红色告警：90%（页面告警） 数据库监控： ├─ 查询延迟：P95 > 1s（黄色） ├─ 查询延迟：P95 > 5s（红色） └─ 连接数：> 80% 最大值（黄色） 6. 级联故障防护：多层降级 Level 1（应用层）： - 快速超时：2 秒 - 熔断机制：5 次失败后打开 Level 2（缓存层）： - 返回过期缓存（可能不准，但能用） - 降低应用刷新频率 Level 3（服务降级）： - 关闭非关键功能 - 返回简化版页面 - 告知用户："维护中，部分功能不可用" Level 4（最后防线）： - 限流：最多处理 X 个请求/秒 - 保护系统不至于完全崩溃 == 改进效果对比 == 改进前 改进后 故障原因 DB 磁盘满 无（自动清理） 故障时间 5 分钟 < 30 秒（或直接预防） 影响用户 100% < 1%（因为分片） 恢复时间 5 分钟 自动恢复 SLA 影响 严重违约 无（自动故障转移） 成本 紧急处理 计划内优化 == 回顾和教训 == 1. 磁盘管理很重要（看似简单但往往被忽视） 2. 应用层超时配置很关键（防止级联） 3. 数据库不能是单点（架构必须优化） 4. 告警阈值要合理（早期预警） 5. 断路器和熔断是必须的（防级联） 6. 自动化处理很重要（人工反应速度慢）

推荐配置：
• RTO 目标：< 3 秒
• RPO 目标：< 1 秒（关键数据），< 1 分钟（普通数据）
• 应用层断路器和熔断机制
• 多路径脑裂检测
• 数据库分片或多实例
