🔥 iptables 五链四表与 ipset 性能优化

文章摘要：本文详细讲解 Linux 防火墙的核心机制 iptables，包括五条数据链、四张规则表的优先级关系，以及如何通过 ipset 实现 100 倍的性能提升。在云平台安全组的实现中，我们积累了大量优化经验，本文分享完整的理论和实战方案。

1. Netfilter 架构

1.1 什么是 Netfilter？

Netfilter 是 Linux 内核中的网络包过滤框架，它在五个关键位置对数据包进行拦截和处理，这五个位置被称为五条链（Chains）。

Netfilter 框架架构
数据包进入系统的完整流程： ┌──────────────────────────────────────────────────┐ │ 网络接口（物理层） │ │ eth0, eth1, ... │ └──────────────────────┬───────────────────────────┘ ↓ 【链 1】PREROUTING（进来之前） ├─ 在系统做路由决策之前 ├─ 对入站数据包进行 DNAT 转换 ├─ 修改目标 IP └─ 例如：外网流量转发到内网虚机 ↓ ┌─────────────────────────────┐ │ 路由决策（Routing Lookup） │ │ 根据目标 IP 决定： │ │ • 转发给本机 → INPUT 链 │ │ • 转发给其他主机 → FORWARD │ └──────┬──────────────────────┘ │ ┌──────┴──────────────┐ ↓ ↓ 【链 2】INPUT 【链 3】FORWARD ├─ 本机数据包 ├─ 转发的数据包 ├─ 到达本机应用 ├─ 经过本机转发 ├─ 例如 SSH、HTTP ├─ 例如虚机流量 └─ 不转发 └─ 受影响最大 │ │ ↓ ↓ ┌────────────┐ ┌────────────┐ │ 本机应用 │ │ 发出去 │ │ (SSH等) │ │ (eth1等) │ └────────────┘ └────────────┘ │ │ └──────┬──────────────┘ ↓ 【链 4】OUTPUT（发出之前） ├─ 本机应用发出的数据包 ├─ 进行 SNAT 转换（修改源 IP） ├─ 例如虚拟 IP 转换 └─ 影响相对较小 ↓ 【链 5】POSTROUTING（发出之后） ├─ 所有离开系统的数据包 ├─ 主要用于 SNAT 转换 ├─ 修改源 IP、源 MAC └─ 例如：虚机 IP 转换为 EIP ↓ ┌──────────────────────────────┐ │ 网络接口（物理层） │ │ 发送出去到网络 │ └──────────────────────────────┘ 数据流向的关键点： • PREROUTING 和 OUTPUT：修改地址的关键点（NAT） • INPUT、FORWARD、OUTPUT：执行防火墙规则的关键点 • POSTROUTING：最后的处理点，确保数据包合法性

1.2 五条链的实际用途

链名 触发��机 应用场景 性能影响
PREROUTING 数据包进入系统立即 DNAT（外网→内网）、负载均衡 ⚠️ 高（影响所有入站包）
INPUT 本机数据包进入 SSH、HTTP 等本机服务访问控制 🟡 中等（只影响本机包）
FORWARD 转发的数据包 虚机间通信、安全组、NAT 转发 ⚠️⚠️ 极高（影响所有业务流量）
OUTPUT 本机应用发出 本机应用的出站过滤 🟡 中等（只影响本机应用）
POSTROUTING 数据包离开前 SNAT（内网→外网）、源 IP 转换 ⚠️ 高（影响所有离开包）

2. 五链详解

2.1 PREROUTING 链

PREROUTING 是入站流量的第一道防线，主要用于 DNAT 转换（目标 IP 转换）。

PREROUTING 链示例：端口转发
场景：外网用户访问 EIP 61.152.10.20:80，需要转发到内网虚机 10.0.1.100:80 # 添加 DNAT 规则 iptables -t nat -A PREROUTING \ -i eth0 \ -d 61.152.10.20 \ -p tcp --dport 80 \ -j DNAT --to-destination 10.0.1.100:80 # 规则分解： # -t nat → 在 NAT 表中操作 # -A PREROUTING → 追加到 PREROUTING 链 # -i eth0 → 进入接口是 eth0 # -d 61.152.10.20 → 目标 IP 是 61.152.10.20 # -p tcp --dport 80 → TCP 协议，目标端口 80 # -j DNAT --to-destination → DNAT 转换动作 # 10.0.1.100:80 → 转换到内网虚机 【规则执行效果】 入站： 源 IP: 1.1.1.1 源 IP: 1.1.1.1 源端口: 12345 源端口: 12345 目标 IP: 61.152.10.20 → 目标 IP: 10.0.1.100 目标端口: 80 目标端口: 80 数据包被 DNAT 转换，发往虚机 返回： 源 IP: 10.0.1.100 源 IP: 61.152.10.20 目标 IP: 1.1.1.1 → 目标 IP: 1.1.1.1 反向 DNAT（由 conntrack 自动处理） 【性能考虑】 • 每个入站包都要经过 PREROUTING • 如果有 100 条 DNAT 规则，每个包要检查 100 条 • 对于高流量场景（PPS > 1M），这是性能瓶颈

2.2 FORWARD 链（最重要！）

FORWARD 链处理所有转发流量，在云平台中这是最关键的链，因为虚机间的所有通信都经过这条链。

FORWARD 链：虚机安全组实现
场景：虚机 10.0.1.10 访问虚机 10.0.2.20，需要检查安全组规则 # 创建虚机专属的安全组链 iptables -N SG_VM_12345_IN # 虚机入站安全组 iptables -N SG_VM_12345_OUT # 虚机出站安全组 # FORWARD 链中引用虚机专属链 iptables -A FORWARD -i vnet12345 -j SG_VM_12345_IN iptables -A FORWARD -o vnet12345 -j SG_VM_12345_OUT # 在虚机专属链中定义规则 # 入站：允许 HTTP iptables -A SG_VM_12345_IN -p tcp --dport 80 -j ACCEPT # 入站：允许 HTTPS iptables -A SG_VM_12345_IN -p tcp --dport 443 -j ACCEPT # 入站：允许 SSH（仅 10.0.0.0/16） iptables -A SG_VM_12345_IN -p tcp --dport 22 \ -m set --match-set sg_vpc_net src -j ACCEPT # 入站：默认拒绝 iptables -A SG_VM_12345_IN -j DROP # 出站：允许所有 iptables -A SG_VM_12345_OUT -j ACCEPT 【执行流程】 虚机 10.0.1.10 → 虚机 10.0.2.20:80 ① 数据包到达 FORWARD 链 ② 匹配出接口规则：-o vnet12345 ✗（出接口不匹配） ③ 匹配入接口规则：-i vnet??? ✗（源是 vnet12345，但不是入） ④ 一条条检查 100 条规则... ⑤ 最后到达 DROP 规则 ⑥ 数据包被丢弃 ❌ 问题：即使规则只有 100 条，每个包都要遍历整个链！ ✓ 解决方案：使用 ipset 将规则从 O(n) 降至 O(1) 【性能数据】 假设有 50 条安全组规则： • 传统 iptables：每个包检查最多 50 条规则 = O(50) • 使用 ipset：直接查表，固定时间 = O(1) 对于 10000 虚机、每虚机 50 条规则、单机 PPS = 1M： • 传统方式：1M × 50 = 50M 条规则检查/秒 • ipset 方式：1M × 1 = 1M 次哈希查询/秒 • 性能提升：50 倍！

2.3 INPUT、OUTPUT、POSTROUTING

其他三条链的典型应用
【INPUT 链】本机服务访问控制 场景：只允许特定 IP 访问本机 SSH iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/16 -j ACCEPT iptables -A INPUT -p tcp --dport 22 -j DROP 作用：保护管理接口，限制 SSH 访问源 【OUTPUT 链】本机应用出站过滤 场景：限制本机应用只能访问特定目标 iptables -A OUTPUT -p tcp -d 8.8.8.8 --dport 53 -j ACCEPT iptables -A OUTPUT -p tcp --dport 53 -j DROP 作用：防止本机恶意应用访问外部服务，或实现数据泄露防护 【POSTROUTING 链】源地址转换（SNAT） 场景：虚机出站流量 NAT 到 EIP iptables -t nat -A POSTROUTING \ -o eth0 \ -s 10.0.0.0/16 \ -j SNAT --to-source 61.152.10.20 作用： • 将虚机私网 IP 转换为公网 IP • 虚机无需公网 IP，只需 EIP • 实现一个公网 IP 供多个虚机使用（连接跟踪）

3. 四表原理

3.1 四张表的执行顺序

iptables 有四张表，每条链的规则都分布在这四张表中。数据包经过每条链时，会依次遍历四张表的规则。

四张表的执行顺序和优先级
数据包经过一条链（例如 FORWARD）时的处理顺序： 进入 FORWARD 链 ↓ 【表 1】raw 表（最先执行） ├─ 功能：设置数据包追踪标志 ├─ 目标：NOTRACK（不跟踪）或 MARK（标记） ├─ 优先级：最高（最先匹配） ├─ 规则数：通常很少（< 10） └─ 例子： iptables -t raw -A FORWARD -p tcp --dport 22 -j NOTRACK ├─ 作用：SSH 连接不进入 conntrack └─ 好处：减少 conntrack 表占用 ↓ 【表 2】mangle 表（其次执行） ├─ 功能：修改数据包头字段（TTL、TOS、MARK） ├─ 目标：MARK（标记）、TTL 修改、QoS ├─ 优先级：第二 ├─ 规则数：中等（10-100） └─ 例子： iptables -t mangle -A FORWARD \ -i vnet* -j MARK --set-mark 0x01 ├─ 作用：标记虚机流量 └─ 好处：后续可用 tc（流量控制）处理 ↓ 【表 3】nat 表（再次执行） ├─ 功能：地址转换（SNAT、DNAT） ├─ 目标：SNAT、DNAT、MASQUERADE ├─ 优先级：第三 ├─ 规则数：中等（10-100） └─ 例子： iptables -t nat -A PREROUTING -d 61.152.10.20 \ -j DNAT --to-destination 10.0.1.100 ├─ 作用：EIP 转换为内网 IP └─ 好处：实现虚机公网访问 ↓ 【表 4】filter 表（最后执行） ├─ 功能：过滤（ACCEPT、DROP、REJECT） ├─ 目标：ACCEPT、DROP、REJECT ├─ 优先级：最低（最后匹配） ├─ 规则数：通常最多（100-10000） └─ 例子： iptables -t filter -A FORWARD -p tcp --dport 80 -j ACCEPT ├─ 作用：允许 HTTP 访问 └─ 好处：安全组控制 ↓ 如果到达这里，数据包被允许 退出链，继续转发或本机处理 【执行顺序总结】 对于任何一条链（PREROUTING、INPUT、FORWARD、OUTPUT、POSTROUTING）： raw → mangle → nat → filter 【性能影响】 假设有： • raw: 5 条规则 • mangle: 50 条规则 • nat: 100 条规则 • filter: 1000 条规则 每个数据包需要检查：5 + 50 + 100 + 1000 = 1155 条规则！ 对于 1M PPS：1M × 1155 = 1.155 亿次规则检查/秒 这就是为什么 ipset 如此重要！

3.2 各表的实际应用

表名 主要功能 典型链 云平台应用
raw 追踪控制 PREROUTING OUTPUT SSH 等不跟踪的协议
mangle QoS、标记 所有链 虚机流量分类、优先级
nat 地址转换 PREROUTING POSTROUTING OUTPUT EIP、SNAT、DNAT、负载均衡
filter 防火墙 INPUT FORWARD OUTPUT 安全组、访问控制

4. 规则匹配流程

4.1 单条规则的完整匹配过程

规则匹配逻辑（逐字段检查）
规则示例： iptables -A FORWARD -i eth0 -p tcp --dport 80 \ -s 10.0.1.0/24 -d 10.0.2.0/24 -j ACCEPT 数据包：源IP 10.0.1.10, 源端口 34567, 目标IP 10.0.2.20, 目标端口 80 匹配过程（逐个条件检查）： ┌─────────────────────────────────┐ │ 条件 1：-i eth0（入接口匹配） │ ├─────────────────────────────────┤ │ 数据包入接口 = eth0？ │ │ ✓ YES → 继续 │ │ ✗ NO → 跳过本规则 │ └─────────────────────────────────┘ ↓ ┌─────────────────────────────────┐ │ 条件 2：-p tcp（协议匹配） │ ├─────────────────────────────────┤ │ 数据包协议 = TCP？ │ │ ✓ YES → 继续 │ │ ✗ NO → 跳过本规则 │ └─────────────────────────────────┘ ↓ ┌─────────────────────────────────┐ │ 条件 3：--dport 80（目标端口） │ ├─────────────────────────────────┤ │ 数据包目标端口 = 80？ │ │ ✓ YES → 继续 │ │ ✗ NO → 跳过本规则 │ └─────────────────────────────────┘ ↓ ┌─────────────────────────────────┐ │ 条件 4：-s 10.0.1.0/24（源 IP） │ ├─────────────────────────────────┤ │ 数据包源 IP 在 10.0.1.0/24？ │ │ ✓ 10.0.1.10 YES → 继续 │ │ ✗ NO → 跳过本规则 │ └─────────────────────────────────┘ ↓ ┌─────────────────────────────────┐ │ 条件 5：-d 10.0.2.0/24（目标IP） │ ├─────────────────────────────────┤ │ 数据包目标 IP 在 10.0.2.0/24？ │ │ ✓ 10.0.2.20 YES → 继续 │ │ ✗ NO → 跳过本规则 │ └─────────────────────────────────┘ ↓ ┌─────────────────────────────────┐ │ ✓ 所有条件匹配！ │ │ 执行目标动作：ACCEPT │ │ 数据包被允许通过 │ └─────────────────────────────────┘ 【关键点】 • 规则的所有条件必须都匹配（AND 逻辑） • 如果有任何条件不匹配，整条规则被跳过 • 继续检查下一条规则 • 直到某条规则匹配或到达链末尾

4.2 链中规则的遍历顺序

链中规则的执行顺序（重要！）
假设 FORWARD 链中有 5 条规则： Rule 1: -p tcp --dport 80 -j ACCEPT Rule 2: -p tcp --dport 443 -j ACCEPT Rule 3: -p tcp --dport 22 -j DROP Rule 4: -j LOG Rule 5: -j DROP 数据包：TCP 连接到 80 端口 执行流程： Rule 1: 检查条件... ├─ -p tcp？ YES ├─ --dport 80？ YES ├─ ✓ 所有条件匹配 ├─ 执行目标：ACCEPT └─ 数据包被接受，退出链 ✓ ❌ Rule 2-5 都不执行，直接跳出链 【关键规则】 • 规则从上到下依次检查 • 第一条匹配的规则决定数据包命运 • 即使后面有相反的规则，也不会被检查 • 因此，规则的顺序非常重要！ 【性能启示】 假设有 1000 条规则： • 常见流量（HTTP）在第 10 条规则匹配 → 需要检查 10 条 • 罕见流量（SSH）在第 900 条规则匹配 → 需要检查 900 条 • 被拒绝流量在最后的 DROP → 需要检查所有 1000 条 解决方案： 1. 将常见规则放在前面 2. 使用 ipset 将规则从 O(n) 降至 O(1) 3. 使用 early drop 快速拒绝已知的坏流量

5. 安全组实现

5.1 安全组的模型

在云平台中，安全组是虚机的网络隔离单位。每个虚机可以关联一个安全组，安全组定义了哪些流量可以进出虚机。

安全组的 iptables 实现
【安全组模型】 虚机 10.0.1.10 关联安全组 "web-sg" 安全组规则： 入站： • 允许 TCP 80（HTTP）来自任何地址 • 允许 TCP 443（HTTPS）来自任何地址 • 允许 TCP 22（SSH）来自 10.0.0.0/16 出站： • 允许所有出站流量 【iptables 实现】 # 步骤 1：创建虚机专属的链 iptables -N SG_10.0.1.10_IN # 入站规则 iptables -N SG_10.0.1.10_OUT # 出站规则 # 步骤 2：在 FORWARD 链中引用虚机专属链 # 对于虚机 10.0.1.10 的入站流量（进入虚机） iptables -I FORWARD 1 -d 10.0.1.10 -j SG_10.0.1.10_IN # 对于虚机 10.0.1.10 的出站流量（离开虚机） iptables -I FORWARD 2 -s 10.0.1.10 -j SG_10.0.1.10_OUT # 步骤 3：在虚机专属链中定义规则 # 入站规则：允许 HTTP iptables -A SG_10.0.1.10_IN -p tcp --dport 80 -j ACCEPT # 入站规则：允许 HTTPS iptables -A SG_10.0.1.10_IN -p tcp --dport 443 -j ACCEPT # 入站规则：允许 SSH（仅 VPC 内） iptables -A SG_10.0.1.10_IN -p tcp --dport 22 \ -s 10.0.0.0/16 -j ACCEPT # 入站规则：允许虚机间通信（如果需要） iptables -A SG_10.0.1.10_IN -s 10.0.0.0/16 -j ACCEPT # 入站规则：默认拒绝 iptables -A SG_10.0.1.10_IN -j DROP # 出站规则：允许所有（通常 Web 服务器需要访问数据库） iptables -A SG_10.0.1.10_OUT -j ACCEPT 【流量的匹配过程】 客户端 (1.1.1.1:12345) → 虚机 (10.0.1.10:80) ① 数据包进入 FORWARD 链 ├─ 检查：-d 10.0.1.10？ YES ├─ 跳转到 SG_10.0.1.10_IN 链 └─ 继续匹配 SG_10.0.1.10_IN 中的规则 ② 在 SG_10.0.1.10_IN 中 ├─ 规则 1：-p tcp --dport 80？ YES ├─ ✓ 匹配！ ├─ 执行 ACCEPT 动作 └─ 数据包被允许，虚机收到 【性能问题】 问题：假设有 10000 虚机，每个虚机 50 条入站规则 总共 500,000 条规则 每个数据包进入 FORWARD 链时： ① 先检查主 FORWARD 链的规则（查询 10000 条虚机目标 IP） ├─ 通过逐条检查或查询 FIB（Forward Information Base） └─ 平均 O(log 10000) ≈ 13 次比较 ② 然后检查目标虚机专属链中的 50 条规则 ├─ O(50) 次规则检查 └─ 如果规则分散，可能需要全部检查 ③ 总耗时：O(13 + 50) = O(63) 次操作 ✓ 优化：使用 ipset 将规则检查从 O(50) 降至 O(1)

6. ipset 原理

6.1 什么是 ipset？

ipset 是 Linux 内核中的IP 集合管理工具，它使用哈希表存储 IP/网络段，使得规则匹配从 O(n) 变为 O(1)。

ipset vs iptables 对比
【传统 iptables 方式】 规则：允许来自 100 个源 IP 的 HTTP 访问 传统方式（100 条规则）： iptables -A FORWARD -p tcp --dport 80 -s 10.0.1.1 -j ACCEPT iptables -A FORWARD -p tcp --dport 80 -s 10.0.1.2 -j ACCEPT iptables -A FORWARD -p tcp --dport 80 -s 10.0.1.3 -j ACCEPT ... (98 条重复规则) 问题： • 规则数量：100 条 • 匹配性能：最坏情况需要检查 100 条 • 更新成本：添加 1 个 IP 需要添加 1 条规则 • 内存占用：每条规则约 200 字节 = 20 KB 【使用 ipset 的方式】 # 创建 ipset 集合（一次性） ipset create http_sources hash:ip hashsize 256 maxelem 10000 # 添加源 IP（直接操作 ipset，不涉及 iptables） ipset add http_sources 10.0.1.1 ipset add http_sources 10.0.1.2 ipset add http_sources 10.0.1.3 ... (添加 100 个 IP) # 单条 iptables 规则（与数量无关） iptables -A FORWARD -p tcp --dport 80 \ -m set --match-set http_sources src -j ACCEPT 优势： • 规则数量：1 条（固定！） • 匹配性能：O(1)，无论有 100 还是 100,000 个 IP • 更新成本：添加 IP 不需要修改 iptables 规则 • 内存占用：哈希表，占用相对固定 性能对比： 场景：10,000 虚机，50 条安全组规则/虚机，总 500,000 条规则 流量：1M PPS（每秒 100 万个数据包） 传统 iptables： • 每个数据包检查 1-50 条规则 • 平均 25 条规则检查 • 总耗时：1M × 25 = 2500 万次规则检查/秒 ipset 优化： • 每个数据包进行 1 次哈希查询 • 哈希查询时间：O(1) ≈ 1 次内存访问 • 总耗时：1M × 1 = 100 万次哈希查询/秒 性能提升：25 倍！

6.2 ipset 的使用

ipset 完整使用指南
# 【类型 1】hash:ip - 存储单个 IP ipset create whitelist hash:ip hashsize 256 maxelem 10000 ipset add whitelist 10.0.1.1 ipset add whitelist 10.0.1.2 ipset test whitelist 10.0.1.1 # 检查 IP 是否在集合中 ipset list whitelist # 列出所有 IP # 【类型 2】hash:net - 存储网络段 ipset create allowed_networks hash:net hashsize 256 maxelem 1000 ipset add allowed_networks 10.0.0.0/16 ipset add allowed_networks 172.16.0.0/12 ipset test allowed_networks 10.0.1.5 # 检查 IP 是否在网络段内 # 【类型 3】hash:ip,port - 存储 IP+端口对 ipset create web_services hash:ip,port hashsize 256 maxelem 10000 ipset add web_services 10.0.1.100,tcp:80 ipset add web_services 10.0.1.100,tcp:443 ipset test web_services 10.0.1.100,tcp:80 # 【在 iptables 中使用】 # 例子 1：匹配源 IP iptables -A FORWARD -m set --match-set whitelist src -j ACCEPT # 例子 2：匹配目标 IP iptables -A FORWARD -m set --match-set allowed_networks dst -j ACCEPT # 例子 3：匹配源 IP+端口 iptables -A FORWARD -m set --match-set web_services src,dst -j ACCEPT # 例子 4：取反（NOT） iptables -A FORWARD -m set ! --match-set whitelist src -j DROP 【性能参数详解】 ipset create http_clients hash:ip \ hashsize 256 \ # 初始哈希表大小（256 个 bucket） maxelem 100000 \ # 最多容纳 100000 个元素 timeout 3600 # 元素自动过期时间（3600 秒） 关键性能参数： • hashsize：影响哈希碰撞概率，越大越快，但占用内存 • maxelem：最大元素数，超过时会自动扩容 • timeout：元素过期时间，节省内存（适合连接跟踪） 【批量操作】 # 从文件批量导入 cat > /tmp/ips.txt << EOF 10.0.1.1 10.0.1.2 10.0.2.0/24 EOF ipset flush http_clients # 清空集合 while read ip; do ipset add http_clients "$ip" done < /tmp/ips.txt # 原子交换（无中断更新） ipset create http_clients_new hash:ip maxelem 100000 while read ip; do ipset add http_clients_new "$ip" done < /tmp/ips.txt ipset swap http_clients http_clients_new # 原子交换，< 1ms ipset destroy http_clients_new 【查看统计信息】 ipset list http_clients # 查看所有元素 ipset list http_clients | grep "^10.0" # 过滤特定元素 ipset list -terse http_clients # 简洁输出

7.2 优化最佳实践

安全组优化最佳实践
【实践 1】使用 ipset 替代长规则链 ❌ 不要这样做（500 条规则）： iptables -A FORWARD -p tcp --dport 80 -s 10.0.1.1 -j ACCEPT iptables -A FORWARD -p tcp --dport 80 -s 10.0.1.2 -j ACCEPT ... (498 条重复) ✓ 应该这样做（1 条规则 + ipset）： ipset create http_sources hash:ip maxelem 10000 for ip in $(cat /tmp/sources.txt); do ipset add http_sources $ip done iptables -A FORWARD -p tcp --dport 80 \ -m set --match-set http_sources src -j ACCEPT 【实践 2】按虚机创建独立的 ipset ✓ 为每个虚机创建专属 ipset： ipset create sg_vm_10.0.1.10_in hash:ip,port maxelem 1000 ipset create sg_vm_10.0.1.10_out hash:ip,port maxelem 1000 iptables -A FORWARD -d 10.0.1.10 \ -m set --match-set sg_vm_10.0.1.10_in dst -j ACCEPT 好处： ├─ 虚机独立隔离 ├─ 易于删除虚机时清理规则 └─ 便于排查特定虚机的网络问题 【实践 3】分层规则：快速拒绝 + 精细允许 ✓ 规则顺序优化： # 层 1：快速拒绝已知坏流量 iptables -A FORWARD -m set --match-set blacklist src -j DROP # 层 2：精细允许白名单 iptables -A FORWARD -m set --match-set whitelist src -j ACCEPT # 层 3：默认拒绝 iptables -A FORWARD -j DROP 好处： ├─ 坏流量快速被拒绝，减少后续规则检查 ├─ 延迟：最坏情况 3 次规则检查 └─ CPU 占用进一步降低 【实践 4】使用原子操作更新规则 ✓ 安全地更新规则（零中断）： ipset create http_sources_new hash:ip maxelem 10000 # 添加新规则... ipset swap http_sources http_sources_new ipset destroy http_sources_new 好处： ├─ 切换时间 < 1ms ├─ 业务流量零中断 └─ 不需要重新加载所有 iptables 规则 【实践 5】监控和告警 ✓ 监控 ipset 使用情况： # 查看元素数量 ipset list http_sources | grep "Number of entries" # 设置告警：如果接近 maxelem，发出告警 if [ $(ipset list http_sources | grep entries | awk '{print $4}') -gt 9000 ]; then echo "WARNING: http_sources ipset approaching max limit" # 发送告警... fi 好处： ├─ 及时发现容量问题 ├─ 提前扩容，避免流量丢失 └─ 性能监控数据 【实践 6】性能调优参数 # ipset 创建时设置最优参数 ipset create http_sources hash:ip \ hashsize 1024 \ # 初始大小 1024（根据预期元素数调整） maxelem 100000 \ # 最多 100000 个元素 timeout 0 # 不自动过期（0 = 永不过期） # iptables 优化 echo "net.ipv4.netfilter.ip_conntrack_max=1000000" >> /etc/sysctl.conf sysctl -p 好处： ├─ 减少哈希碰撞，提升查询速度 ├─ 充足的元素容量，避免扩容开销 └─ 规则缓存优化 【实践 7】定期清理过期规则 ✓ 清理不活跃的 IP： # 假设使用了 timeout（过期时间） ipset create http_sources hash:ip maxelem 100000 timeout 3600 # 查看包含过期时间的元素 ipset list -terse http_sources | tail -20 # 如果需要手动清理 ipset del http_sources 10.0.1.1 好处： ├─ 节省内存 ├─ 加快哈希查询（减少元素数量） └─ 定期维护，保持系统整洁

8.1 完整的生产级安全组配置

生产级安全组脚本（Python）
#!/usr/bin/env python3 """ 云平台安全组管理脚本 使用 ipset + iptables 实现高性能安全组 """ import subprocess import json import sys from typing import List, Dict, Set class SecurityGroupManager: def __init__(self): self.config_file = "/etc/security-groups.json" self.sg_config = {} self.ipset_prefix = "sg_" def load_config(self): """加载安全组配置""" try: with open(self.config_file, 'r') as f: self.sg_config = json.load(f) print(f"✓ Loaded config: {len(self.sg_config)} security groups") except Exception as e: print(f"✗ Failed to load config: {e}") sys.exit(1) def run_command(self, cmd): """执行系统命令""" try: result = subprocess.run(cmd, shell=True, capture_output=True) if result.returncode != 0: print(f"✗ Command failed: {cmd}") print(f" Error: {result.stderr.decode()}") return False return True except Exception as e: print(f"✗ Exception: {e}") return False def create_ipsets(self): """为每个安全组创建 ipset""" print("\n=== Creating ipsets ===") for sg_name, sg_rules in self.sg_config.items(): for direction in ["in", "out"]: ipset_name = f"{self.ipset_prefix}{sg_name}_{direction}" # 删除旧的 ipset（如果存在） self.run_command(f"ipset destroy {ipset_name} 2>/dev/null") # 创建新 ipset cmd = f"ipset create {ipset_name} hash:ip,port maxelem 100000" if self.run_command(cmd): print(f"✓ Created ipset: {ipset_name}") def populate_ipsets(self): """填充 ipset 规则""" print("\n=== Populating ipsets ===") for sg_name, sg_rules in self.sg_config.items(): # 入站规则 for rule in sg_rules.get("ingress", []): protocol = rule.get("protocol", "tcp") port = rule.get("port", 0) source = rule.get("source", "0.0.0.0/0") ipset_name = f"{self.ipset_prefix}{sg_name}_in" entry = f"{source},{protocol}:{port}" cmd = f"ipset add {ipset_name} {entry} 2>/dev/null" self.run_command(cmd) # 出站规则 for rule in sg_rules.get("egress", []): protocol = rule.get("protocol", "tcp") port = rule.get("port", 0) destination = rule.get("destination", "0.0.0.0/0") ipset_name = f"{self.ipset_prefix}{sg_name}_out" entry = f"{destination},{protocol}:{port}" cmd = f"ipset add {ipset_name} {entry} 2>/dev/null" self.run_command(cmd) def apply_iptables_rules(self): """应用 iptables 规则""" print("\n=== Applying iptables rules ===") # 清空 FORWARD 链的自定义规则 self.run_command("iptables -F FORWARD") for sg_name, sg_rules in self.sg_config.items(): ipset_in = f"{self.ipset_prefix}{sg_name}_in" ipset_out = f"{self.ipset_prefix}{sg_name}_out" # 虚机 IP（从配置获取） vm_ip = sg_rules.get("vm_ip") if not vm_ip: continue # 入站规则：目标 IP = 虚机 IP，源 IP 在 ipset 中 cmd = f"iptables -A FORWARD -d {vm_ip} " \ f"-m set --match-set {ipset_in} src,dst -j ACCEPT" self.run_command(cmd) print(f"✓ Applied ingress rule for {vm_ip}") # 出站规则：源 IP = 虚机 IP，目标 IP 在 ipset 中 cmd = f"iptables -A FORWARD -s {vm_ip} " \ f"-m set --match-set {ipset_out} dst,src -j ACCEPT" self.run_command(cmd) print(f"✓ Applied egress rule for {vm_ip}") # 默认拒绝 self.run_command("iptables -A FORWARD -j DROP") print("✓ Applied default DROP rule") def apply(self): """应用所有配置""" self.load_config() self.create_ipsets() self.populate_ipsets() self.apply_iptables_rules() print("\n✓ Security groups applied successfully!") def status(self): """查看当前状态""" print("\n=== Security Group Status ===") # 显示 ipset print("\nipsets:") self.run_command("ipset list -n | grep sg_") # 显示 iptables 规则 print("\niptables FORWARD chain:") self.run_command("iptables -L FORWARD -v -n") def cleanup(self): """清理所有规则""" print("\n=== Cleaning up ===") # 删除所有 sg_ 开头的 ipset result = subprocess.run("ipset list -n | grep ^sg_", shell=True, capture_output=True) for ipset_name in result.stdout.decode().split('\n'): if ipset_name: self.run_command(f"ipset destroy {ipset_name}") # 清空 FORWARD 链 self.run_command("iptables -F FORWARD") print("✓ Cleanup completed") if __name__ == "__main__": manager = SecurityGroupManager() if len(sys.argv) > 1: action = sys.argv[1] if action == "apply": manager.apply() elif action == "status": manager.status() elif action == "cleanup": manager.cleanup() else: print(f"Unknown action: {action}") else: print("Usage: python3 sg_manager.py [apply|status|cleanup]")
