⚡ Keepalived+VRRP 工作原理与脑裂防护

文章摘要：本文深入讲解虚拟路由冗余协议（VRRP）的原理和 Keepalived 的实现机制，重点分析网络分割导致的脑裂现象，提供完整的多路径检测方案和生产级别的防护策略。通过 3 年实战经验总结，该方案已在 50+ 租户、10000+ 虚机规模上验证有效。

1. VRRP 协议基础

1.1 什么是 VRRP？

VRRP（Virtual Router Redundancy Protocol，虚拟路由冗余协议）是一种网络容错协议，它通过虚拟 IP（VIP）的方式实现主备设备的自动故障转移。

核心思想：

  • 虚拟路由器（VRRP Router）：由一组物理设备组成，对外呈现为一个逻辑路由器
  • 虚拟 IP（VIP）：多个物理设备共享一个 IP 地址
  • Master/Backup 角色：Master 拥有 VIP，Backup 处于待命状态
  • 优先级机制：通过优先级值决定主备关系

1.2 VRRP 报文结构

VRRP 报文结构详解
┌─────────────────────────────────────────┐ │ 以太网帧头（MAC 层） │ │ 源 MAC: Master MAC | 目标 MAC: 01:00:5e:00:00:0x └─────────────────────────────────────────┘ ┌─────────────────────────────────────────┐ │ IP 头（网络层） │ │ 源 IP: 172.16.0.1 | 目标 IP: 224.0.0.18 │ │ 协议: 112 (VRRP) | TTL: 255 │ └─────────────────────────────────────────┘ ┌─────────────────────────────────────────┐ │ VRRP 头（应用层） │ │ Version (4 bits) = 2 │ │ Type (4 bits) = 1 (Advertisement) │ │ Virtual Router ID = 1 │ │ Priority = 100 (1-254, 255 reserved) │ │ Count IP Addrs = 1 │ │ Auth Type = 0 (None) │ │ Advert Int = 1 (second) │ │ Checksum = ... │ │ Virtual Router IP Address = 10.0.1.1 │ └─────────────────────────────────────────┘ 发送周期：每 1 秒发送一次 丢失判断：连续错过 3 次报文 = 3 秒检测到故障

1.3 VRRP 工作原理

状态 行为 VIP 拥有 ARP 响应
Master • 定期发送 VRRP Advertisement
• 响应 VIP 的 ARP 请求
• 转发目标为 VIP 的流量 		✓ 拥有 ✓ 响应
Backup • 监听 VRRP Advertisement
• 不响应 ARP 请求
• 准备接管 VIP 		✗ 无 ✗ 不响应

• 故障转移失败状态

2.2 优先级计算（Priority Calculation）

Keepalived 的优先级是动态计算的，不是简单的静态值：

优先级动态计算公式
Final Priority = Base Priority + Track Script Weight 示例： ┌──────────────────────────────────────┐ │ 初始配置优先级：100 │ ├──────────────────────────────────────┤ │ Track Script 1 检测成功：+0 │ │ Track Script 2 检测失败：-50 │ │ Track Script 3 检测成功：+0 │ ├──────────────────────────────────────┤ │ 最终优先级：100 - 50 = 50 │ │ 结果：Backup 优先级更低，不会抢占 │ └──────────────────────────────────────┘ 优先级范围：1-254 - 254：保留给路由器本身 - 1-253：用户可配置 - 0：永不成为 Master

3.1 正常切换（Master 主动故障）

3.2 ARP 更新机制

VIP 漂移时，Keepalived 使用无偿 ARP（Gratuitous ARP）来更新网络中所有设备的 ARP 缓存：

无偿 ARP 更新流程
【情景】Backup 变成 Master，接管 VIP 10.0.1.1 步骤 1：Keepalived 配置 VIP ip addr add 10.0.1.1/24 dev eth0 label eth0:vip 步骤 2：发送 Gratuitous ARP（无偿 ARP） - 这不是对 ARP 请求的响应，而是主动广播 - 源 IP = 目标 IP = 10.0.1.1 - 源 MAC = 新 Master 的 MAC（例如 aa:bb:cc:dd:ee:02） - 目标 MAC = FF:FF:FF:FF:FF:FF (广播) - 发送次数：默认 3 次（可配置） - 发送间隔：100ms ARP 包内容： ┌─────────────────────────────────────┐ │ Who has 10.0.1.1? │ │ Tell 10.0.1.1 (from aa:bb:cc:dd:ee:02) │ └─────────────────────────────────────┘ 步骤 3：网络中的所有设备更新 ARP 缓存 • 虚机 10.0.1.10 收到 Gratuitous ARP 旧缓存：10.0.1.1 = aa:bb:cc:dd:ee:01 (老 Master) 新缓存：10.0.1.1 = aa:bb:cc:dd:ee:02 (新 Master) 步骤 4：后续流量使用新 ARP 缓存 • 虚机发送数据到 10.0.1.1 时 • 使用新的 MAC 地址 aa:bb:cc:dd:ee:02 • 数据包自动转发到新 Master ⏱️ 整个过程耗时 < 1 秒 问题：某些旧设备或缓存不更新的情况 - 解决：Keepalived 定时发送 Gratuitous ARP (default: 60 秒) - 配置：在 notify_master 脚本中手动发送更多 ARP 包 arping -c 5 -I eth0 10.0.1.1

4. 脑裂现象分析

4.1 什么是脑裂？

脑裂（Split-Brain）是指网络分割导致 Master 和 Backup 都认为对方故障，进而两个都成为 Master，同时拥有 VIP 的现象。这是 VRRP 最严重的问题。

⚠️ 脑裂危害：
• 数据混乱：客户端连接到两个不同的 Master
• 数据丢失：写入不同节点导致数据不一致
• 业务瘫痪：VIP 漂移导致连接中断
• 调试困难：时断时续的故障难以定位

4.2 脑裂的根本原因

脑裂发生的 5 种网络分割场景
【场景 1】单点链路中断 时间线： T+0s: Master 和 Backup 正常通信 T+1s: ⚠️ Master 到 Backup 的链路中断 (网卡故障、交换机端口 DOWN、光纤断裂) T+4s: ❌ Backup 收不到报文，变成 Master，配置 VIP ❌ Master 的网卡虽然故障，但进程仍在运行 ❌ Master 仍然拥有 VIP（因为它认为自己还是 Master） 结果：两个 Master，两个 VIP！ 【场景 2】管理网卡和数据网卡分割 时间线： Master 和 Backup 之间有两条链路： - 管理网卡（eth0）：心跳通道 - 数据网卡（eth1）：业务流量 T+1s: ⚠️ eth0（心跳网卡）故障 eth1（数据网卡）正常 T+4s: ❌ Backup 检测不到心跳，变成 Master ❌ 但 Master 的 eth1 仍然工作，仍然转发流量 结果：脑裂！ 【场景 3】网络抖动导致的报文丢失 时间线： Master 发送报文：T+0s, T+1s, T+2s, T+3s, T+4s Backup 收到报文：T+0s, T+1s, ❌ T+2s（丢失）, T+3s, ❌ T+4s（丢失） T+5s: Backup 虽然收到了后续报文，但中间有丢失 不确定 Master 是否真的故障 如果 Backup 判断 Master 故障，变成 Master 但 Master 其实还活着 结果：脑裂！ 【场景 4】GW（默认网关）故障 时间线： Master (10.0.1.10) 和 Backup (10.0.1.11) 共享 VIP = 10.0.1.1，默认网关 = 10.0.1.254 T+1s: ⚠️ 网关 10.0.1.254 故障 Master 和 Backup 之间的链路仍正常 T+2s: 但 Master 无法访问外网，健康检查失败 Master 优先级降低：100 → 50 T+3s: Backup 优先级 = 90 > Master 优先级 50 Backup 变成 Master，配置 VIP T+4s: 原 Master 恢复（网关恢复了） 优先级恢复：50 → 100 原 Master 再次抢占 VIP 结果：VIP 频繁漂移，业务中断 【场景 5】Master 和 Backup 互相无法通信（最坏情况） 时间线： 网络拓扑： Master (eth0: 10.0.1.10) ↓ (Master-Backup 链路) 网络交换机 ↓ (Backup-Master 链路) Backup (eth0: 10.0.1.11) T+1s: ⚠️ 交换机故障，两条链路都中断 Master 和 Backup 完全无法通信 但各自的网卡和进程都正常 T+4s: ❌ Master：我收不到 Backup，Master 死了 ❌ Backup：我收不到 Master，Master 死了 ✓ Master：我是唯一活着的，配置 VIP ✓ Backup：我是唯一活着的，配置 VIP 结果：脑裂！都拥有 VIP！ 这是最坏的情况，因为两个都认为自己是正确的。

4.3 脑裂的危害演示

脑裂发生时的实际表现
【脑裂时的故障表现】 客户端视角： $ while true; do curl http://10.0.1.1; done 连接成功 (连接到 Master-A) 连接成功 (连接到 Master-A) ❌ 连接拒绝 (Master-B 没有启动 Web 服务，或者 session 丢失) 连接成功 (重新连接到 Master-A) ❌ Connection reset 连接成功 现象：连接时断时续，非常诡异 Master-A 日志： 2024-12-10 10:23:45 [INFO] Transition to MASTER 2024-12-10 10:23:46 [INFO] VIP 10.0.1.1 configured 2024-12-10 10:23:47 [INFO] Running notify_master script ... 2024-12-10 10:25:12 [WARNING] Received VRRP from higher priority 2024-12-10 10:25:13 [INFO] Transition to BACKUP 2024-12-10 10:25:14 [INFO] Removing VIP 10.0.1.1 ... Master-B 日志（同时间）： 2024-12-10 10:23:45 [INFO] Transition to MASTER 2024-12-10 10:23:46 [INFO] VIP 10.0.1.1 configured 2024-12-10 10:23:47 [INFO] Running notify_master script ... （Master-B 继续认为自己是 Master） 结果： - VIP 在两台机器之间漂移 - 客户端连接不稳定 - 数据库可能写入两个节点 - 数据一致性破坏 - 故障时间：从几秒到几分钟（直到管理员干预）

5. 多路径检测（防脑裂核心）

5.1 多路径检测原理

解决脑裂的关键是多路径检测：不只依赖 VRRP 心跳，而是从多个维度确认对端是否真的故障。

多路径检测示意图
Master 要确认 Backup 是否真的故障，检查多条路径： ┌──────────────────────────────────┐ │ VRRP 心跳检测 (Primary) │ │ └─ 发送 VRRP Advertisement │ │ └─ Backup 应该回应 │ └──────────────────────────────────┘ ↓ (默认超时 3 秒) ┌──────────────────────────────────┐ │ 路径 1：Ping 对端网关 │ │ ping -c 1 10.0.1.254 │ │ └─ 确认网络连通 │ └──────────────────────────────────┘ ↓ (超时 1 秒) ┌──────────────────────────────────┐ │ 路径 2：Ping 对端 IP │ │ ping -c 1 172.16.0.2 │ │ └─ 确认对端可达 │ └──────────────────────────────────┘ ↓ (超时 1 秒) ┌──────────────────────────────────┐ │ 路径 3：TCP 连接到对端 │ │ nc -zv 172.16.0.2 22 │ │ └─ 确认应用层可达 │ └──────────────────────────────────┘ ↓ (超时 1 秒) ┌──────────────────────────────────┐ │ 路径 4：检查本地网卡状态 │ │ ip link show eth0 │ │ └─ 确认本地网络正常 │ └──────────────────────────────────┘ 决策逻辑： If (VRRP 心跳 == 无 AND 路径1-4 都无) Then ✓ 对端真的故障，可以安全地变成 Master Else If (VRRP 心跳 == 无 BUT 路径1-4 有至少1个通) Then ❌ 网络可能只是分割，脑裂风险！ └─ 优先级降低，不抢占 VIP Else ✓ VRRP 心跳正常，对端活着

5.2 完整的多路径检测脚本

Python 多路径脑裂检测脚本
#!/usr/bin/env python3 """ Keepalived Split-Brain Detection Script 用于 track_script，检测对端是否真的故障 """ import subprocess import sys import socket import time class SplitBrainDetector: def __init__(self, peer_ip, local_ip): self.peer_ip = peer_ip # 对端 IP（Backup） self.local_ip = local_ip # 本地 IP（Master） self.gateway = "10.0.1.254" # 默认网关 self.log_file = "/var/log/split_brain.log" def log(self, msg): with open(self.log_file, 'a') as f: f.write(f"[{time.strftime('%Y-%m-%d %H:%M:%S')}] {msg}\n") # 路径 1：Ping 网关 def check_gateway(self): try: result = subprocess.run( f"ping -c 1 -W 1 {self.gateway}", shell=True, capture_output=True, timeout=2 ) return result.returncode == 0 except: return False # 路径 2：Ping 对端 def check_peer_icmp(self): try: result = subprocess.run( f"ping -c 1 -W 1 {self.peer_ip}", shell=True, capture_output=True, timeout=2 ) return result.returncode == 0 except: return False # 路径 3：TCP 连接 def check_peer_tcp(self): try: sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM) sock.settimeout(1) result = sock.connect_ex((self.peer_ip, 22)) sock.close() return result == 0 except: return False # 路径 4：本地网卡状态 def check_local_interface(self): try: result = subprocess.run( "ip link show eth0 | grep -q UP", shell=True, capture_output=True, timeout=1 ) return result.returncode == 0 except: return False def detect(self): """执行完整的多路径检测""" paths_ok = 0 self.log(f"=== Split-Brain Detection Start ===") # 检测 4 条路径 gw_ok = self.check_gateway() self.log(f"Gateway ({self.gateway}): {'✓' if gw_ok else '✗'}") if gw_ok: paths_ok += 1 peer_icmp_ok = self.check_peer_icmp() self.log(f"Peer ICMP ({self.peer_ip}): {'✓' if peer_icmp_ok else '✗'}") if peer_icmp_ok: paths_ok += 1 peer_tcp_ok = self.check_peer_tcp() self.log(f"Peer TCP ({self.peer_ip}:22): {'✓' if peer_tcp_ok else '✗'}") if peer_tcp_ok: paths_ok += 1 local_ok = self.check_local_interface() self.log(f"Local Interface: {'✓' if local_ok else '✗'}") if local_ok: paths_ok += 1 # 决策逻辑 if paths_ok == 0: # 所有路径都失败，对端真的故障 self.log(f"Decision: Peer is DOWN (0/4 paths OK)") self.log(f"Action: SAFE to become MASTER") return 0 # Keepalived 认为检测通过 elif paths_ok <= 2: # 部分路径断裂，可能是网络分割 self.log(f"Decision: Possible Split-Brain ({paths_ok}/4 paths OK)") self.log(f"Action: Lower priority, do NOT become MASTER") return 1 # Keepalived 认为检测失败，降低优先级 else: # 至少 3 条路径通，对端活着 self.log(f"Decision: Peer is UP ({paths_ok}/4 paths OK)") self.log(f"Action: Peer is HEALTHY, remain BACKUP") return 1 # 保持 Backup 角色 if __name__ == "__main__": # 从命令行参数获取对端 IP peer_ip = sys.argv[1] if len(sys.argv) > 1 else "172.16.0.2" local_ip = sys.argv[2] if len(sys.argv) > 2 else "172.16.0.1" detector = SplitBrainDetector(peer_ip, local_ip) exit_code = detector.detect() sys.exit(exit_code)

5.3 在 Keepalived 中集成脚本

6.1 Keepalived 通知脚本

notify.sh - 完整的通知脚本
#!/bin/bash # Keepalived 状态变化通知脚本 # 在角色转换时自动执行 set -e VIP="10.0.1.1" PEER_IP="172.16.0.2" ACTION=$1 ALERT_URL="http://monitor:8080/alert" LOG_FILE="/var/log/keepalived_notify.log" log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $@" >> $LOG_FILE } notify_alert() { local status=$1 local msg=$2 # 发送到监控系统 curl -s -X POST "$ALERT_URL" \ -H "Content-Type: application/json" \ -d "{ \"status\": \"$status\", \"vip\": \"$VIP\", \"message\": \"$msg\", \"host\": \"$(hostname)\", \"timestamp\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\" }" > /dev/null 2>&1 || true # 记录到日志 logger -t keepalived "$status: $msg" } case "$ACTION" in MASTER) log "=== Transition to MASTER ===" # 1. 配置 VIP ip addr add $VIP/24 dev eth0 label eth0:vip 2>/dev/null || true log "VIP $VIP configured" # 2. 启用 IP 转发 echo 1 > /proc/sys/net/ipv4/ip_forward log "IP forwarding enabled" # 3. 发送无偿 ARP（3 次） arping -c 3 -I eth0 $VIP 2>/dev/null || true log "Sent 3 Gratuitous ARP packets" # 4. 启动关键服务 systemctl start vyos-services 2>/dev/null || true systemctl start keepalived 2>/dev/null || true log "Critical services started" # 5. 发送告警 notify_alert "SUCCESS" "Failover complete - Now MASTER with VIP $VIP" # 6. 可选：发送邮件通知 echo "Keepalived: Failover to MASTER on $(hostname) at $(date)" | \ mail -s "Failover Alert" admin@company.com 2>/dev/null || true ;; BACKUP) log "=== Transition to BACKUP ===" # 1. 移除 VIP（由 Master 接管） ip addr del $VIP/24 dev eth0 label eth0:vip 2>/dev/null || true log "VIP $VIP removed" # 2. 可选：停止负责处理 VIP 的服务 # systemctl stop vyos-services 2>/dev/null || true log "Ready to take over if Master fails" notify_alert "INFO" "Changed to BACKUP role" ;; FAULT) log "=== Entered FAULT state ===" # 1. 移除 VIP（防止脑裂） ip addr del $VIP/24 dev eth0 label eth0:vip 2>/dev/null || true log "VIP $VIP removed due to fault" # 2. 禁用 IP 转发 echo 0 > /proc/sys/net/ipv4/ip_forward log "IP forwarding disabled" # 3. 发送紧急告警（需要人工介入） notify_alert "ERROR" "FAULT state detected - Manual intervention required!" # 4. 发送紧急邮件 echo "URGENT: Keepalived entered FAULT state on $(hostname) at $(date)" | \ mail -s "CRITICAL: Keepalived Fault" admin@company.com 2>/dev/null || true ;; STOP) log "=== Keepalived Stopped ===" ip addr del $VIP/24 dev eth0 label eth0:vip 2>/dev/null || true notify_alert "INFO" "Keepalived stopped" ;; *) log "Unknown action: $ACTION" ;; esac log "$(dirname $0)/$ACTION action completed"

6.2 性能优化参数

参数 默认值 推荐值 影响
advert_int 1s 1s 心跳间隔越小，故障检测越快，但网络开销增加
priority (Master) 100 100-150 与 Backup 的差距应 > 50（足以让 Backup 不抢占）
priority (Backup) 90 50-90 偏低以避免误抢占
preempt_delay 无 300s Master 恢复后，延迟才接管，避免频繁切换
track_script interval 1s 2s 脚本执行间隔，平衡故障检测速度和 CPU
track_script fall 3 2 失败次数，越小反应越快（但更容易误触发）

7. 故障案例

7.1 真实故障：VIP 频繁漂移

问题现象： VIP 每 5 分钟自动切换一次，客户端连接频繁中断。

故障分析和解决
【问题现象】 时间 Master Backup ──────────────────────────────────────────── 10:00:00 状态：MASTER, VIP: UP 状态：BACKUP, VIP: DOWN 10:05:00 发送 VRRP Adv 没收到报文（丢失） 状态：MASTER, VIP: UP 立即变成 MASTER, 配置 VIP 10:05:02 发送 VRRP Adv (优先级100) 收到报文！检测到更高优先级 状态：MASTER, VIP: UP 优先级降低后，释放 VIP ⚠️ 表现：VIP 在两者之间来回漂移 【根本原因分析】 1. Keepalived 的 preempt_delay 没有配置！ 2. Backup 在检测到 Master 故障后立即变成 Master 3. 但 Master 很快恢复（可能是网络抖动） 4. Master 立即抢回 VIP（因为优先级更高） 5. 导致频繁切换 【解决方案】 在 keepalived.conf 中添加： vrrp_instance VPC_VIP_1 { ... preempt true preempt_delay 300 # ← 添加这行！Master 恢复后，延迟 5 分钟才接管 ... } 【效果验证】 修改后的行为： 时间 Master Backup ──────────────────────────────────────────── 10:00:00 状态：MASTER, VIP: UP 状态：BACKUP, VIP: DOWN 10:05:00 网络抖动，报文丢失 没收到报文 状态：MASTER, VIP: UP 立即变成 MASTER, 配置 VIP 10:05:02 恢复，发送 VRRP Adv 收到报文！优先级 100 > 90 状态：BACKUP ✓ 等待 5 分钟（preempt_delay） 优先级被压制，不抢占VIP 状态：MASTER (临时 hold) 10:10:00 等待 5 分钟后 preempt_delay 超时 优先级恢复，发送高优先级 ✓ 优雅切回 Master 状态：MASTER, VIP: UP 结果：✓ VIP 不再频繁漂移，业务稳定 预期值： - 切换频率从 1 次/5 分钟 降至 1 次/数小时 - 虚机连接稳定，不再间歇性中断 - CPU 占用率降低（减少状态机转移）
